❓ Motivación

El Problema de la Búsqueda Tradicional

Las bases de datos tradicionales buscan por coincidencia exacta de texto. Pero el lenguaje humano es ambiguo, y los sinónimos son la norma.

❌ Búsqueda Tradicional (SQL LIKE)

SQL 
SELECT * FROM productos
WHERE descripcion LIKE '%zapato%';

-- ❌ NO encuentra:
-- "calzado deportivo"
-- "zapatillas para correr"
-- "tennis Nike Air"

Solo encuentra coincidencias exactas de texto

✅ Búsqueda Vectorial (Semántica)

Concepto 
# Buscar: "zapato"
# ✅ Encuentra por SIGNIFICADO:

"calzado deportivo"     # similitud: 0.92
"zapatillas para correr" # similitud: 0.89
"tennis Nike Air"      # similitud: 0.85
"botas de montaña"     # similitud: 0.78

Encuentra documentos por significado semántico

🔢 Concepto Clave

¿Qué son los Embeddings?

Un embedding es una representación numérica (vector) de un dato. Textos con significado similar producen vectores cercanos en el espacio.

Python — Generando Embeddings 
from langchain_openai import OpenAIEmbeddings

# Crear modelo de embeddings
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small"
)

# Convertir texto → vector
vector = embeddings.embed_query(
    "Gato persa dormido"
)

print(len(vector))  # → 1536
print(vector[:5])
# → [0.012, -0.034, 0.078, ...]

📐 Métricas

Métricas de Distancia

Para encontrar vectores "similares", necesitamos medir la distancia entre ellos. Hay tres métricas principales que pgvector soporta nativamente.

📏
Distancia Coseno
1 - cos(θ)
Mide el ángulo entre dos vectores. Ignora la magnitud, solo se fija en la dirección. La más popular para texto.
Operador pgvector: <=>
📐
Distancia Euclidiana
√Σ(aᵢ - bᵢ)²
Mide la distancia "en línea recta" entre dos puntos en el espacio. Considera la magnitud de los vectores.
Operador pgvector: <->
✖️
Producto Interno
Σ(aᵢ × bᵢ)
Combina dirección y magnitud. Valores más altos = más similares. Útil cuando la magnitud importa (popularidad).
Operador pgvector: <#>

💡 ¿Cuál usar? Para textos con embeddings de OpenAI, la similitud coseno es la opción recomendada. Los vectores ya están normalizados.

🐘 PostgreSQL

pgvector: Vectores en PostgreSQL

pgvector es una extensión que agrega soporte nativo para vectores a PostgreSQL. No necesitas otra base de datos — tu PostgreSQL ya puede ser vectorial.

SQL — Crear tabla con vectores 
-- 1. Habilitar la extensión
CREATE EXTENSION IF NOT EXISTS vector;

-- 2. Crear tabla con columna vector
CREATE TABLE documentos (
    id      SERIAL PRIMARY KEY,
    texto   TEXT,
    metadata JSONB,
    embedding vector(1536)  -- ← tipo vector!
);

-- 3. Crear índice para búsquedas rápidas
CREATE INDEX ON documentos
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- 4. Buscar los 5 más similares
SELECT texto, 1 - (embedding <=> '[0.1, 0.2, ...]')
    AS similitud
FROM documentos
ORDER BY embedding <=> '[0.1, 0.2, ...]'
LIMIT 5;
✅

¿Por qué pgvector?

No necesitas infraestructura extra. Si ya usas PostgreSQL, solo agregas la extensión. SQL + vectores en un solo lugar.

⚡

Índices Optimizados

IVFFlat: rápido, buen recall. HNSW: mayor recall, más memoria. Ambos escalan a millones de vectores.

🔗

Filtrado Híbrido

Combina búsqueda vectorial con filtros SQL tradicionales: WHERE categoria = 'tech' ORDER BY embedding <=> query

🐘 ¿Por qué elegir pgvector?

  • 💰 Sin costo extra: Si ya tienes PostgreSQL, solo agregas la extensión. No pagas por otro servicio.
  • 🔗 SQL + Vectores: Combina queries tradicionales con búsqueda semántica en una sola consulta.
  • 🏗️ ACID & Backups: Hereda toda la robustez de PostgreSQL: transacciones, replicación, pg_dump.
  • 🚀 Simplicidad: Un solo servicio para datos relacionales + vectoriales. Menos infraestructura = menos problemas.

🔗 Integración

LangChain + pgvector

Python — Conexión al VectorStore 
from langchain_openai import OpenAIEmbeddings
from langchain_postgres import PGVector

# 1. Modelo de embeddings
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 2. Conexión a PostgreSQL con pgvector
vectorstore = PGVector(
    embeddings=embeddings,
    collection_name="mi_coleccion",
    connection="postgresql+psycopg://postgres:postgres@localhost:5432/vectordb",
    use_jsonb=True,
)

print("✅ Conectado a pgvector!")
Python — Insertar documentos 
from langchain_core.documents import Document

# Documentos con texto + metadata
docs = [
    Document(
        page_content="Python es un lenguaje de programación...",
        metadata={"categoria": "lenguajes", "año": 2024}
    ),
    Document(
        page_content="FastAPI es un framework moderno...",
        metadata={"categoria": "frameworks", "año": 2024}
    ),
]

# Insertar → vectoriza automáticamente
ids = vectorstore.add_documents(docs)
print(f"Insertados: {len(ids)} docs")
Python — Búsqueda semántica 
# Búsqueda por similitud
resultados = vectorstore.similarity_search(
    "frameworks para APIs web",
    k=3  # Top 3 resultados
)

for doc in resultados:
    print(doc.page_content[:80])

# Búsqueda con scores
resultados = vectorstore.similarity_search_with_score(
    "frameworks para APIs web", k=3
)
for doc, score in resultados:
    print(f"[{score:.4f}] {doc.page_content[:60]}")

# Búsqueda con filtro de metadata
resultados = vectorstore.similarity_search(
    "herramientas IA", k=3,
    filter={"categoria": "ia"}
)
🤖 RAG

RAG: Retrieval-Augmented Generation

La combinación definitiva: buscar en tu base de datos vectorizada y usar los documentos encontrados como contexto para el LLM.

🔄 Flujo RAG Completo

👤 Pregunta "¿Qué es Docker?"
→
🔍 Retriever pgvector search
→
📝 Prompt Pregunta + Contexto
→
🤖 LLM GPT-4o-mini
→
💬 Respuesta Basada en docs
Python — Cadena RAG con LCEL 
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough

# Retriever desde pgvector
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# Prompt con contexto
prompt = ChatPromptTemplate.from_messages([
    ("system", "Responde basándote en:\n{contexto}"),
    ("human", "{pregunta}")
])

# Cadena RAG completa
cadena = (
    {"contexto": retriever | format_docs, "pregunta": RunnablePassthrough()}
    | prompt | modelo | StrOutputParser()
)

respuesta = cadena.invoke("¿Qué es Docker?")
# → Respuesta basada en TUS documentos 🎉
🧠 Avanzado

RAG con Memoria Conversacional

Combina búsqueda vectorial + memoria con LangGraph. El agente recuerda la conversación Y busca en la base de datos.

Python — RAG + Memoria (LangGraph) 
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver

# Estado con mensajes + contexto
class EstadoRAG(TypedDict):
    messages: Annotated[list, add_messages]
    contexto: str

# Nodo 1: Buscar documentos
def recuperar(estado):
    docs = retriever.invoke(
        estado["messages"][-1].content
    )
    return {"contexto": format_docs(docs)}

# Nodo 2: Responder con contexto
def responder(estado):
    respuesta = modelo.invoke(
        system + estado["messages"]
    )
    return {"messages": [respuesta]}

# Compilar con memoria
app = grafo.compile(
    checkpointer=MemorySaver()
)
🔍

Retriever

Busca en pgvector los documentos más relevantes para cada pregunta del usuario.

🧠

Memoria

MemorySaver persiste el historial de mensajes. El agente recuerda toda la conversación.

💬

Seguimiento

El usuario puede hacer preguntas de seguimiento: "¿Y cómo se conecta eso con lo anterior?"

💡 thread_id aísla conversaciones. Igual que un chat individual: cada thread tiene su propia memoria.

⚡ Performance

Índices para Búsqueda Rápida

Sin índices, pgvector hace búsqueda lineal (escanea todos los vectores). Con índices, las búsquedas son sub-lineales y escalan a millones de registros.

📊

IVFFlat

Inverted File Index con cuantización plana. Divide el espacio en clusters (listas) y busca solo en los más cercanos.

SQL 
CREATE INDEX ON documentos
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- lists ≈ √N (N = total de filas)
-- Más listas = más rápido pero menos recall
  • ✅ Creación rápida del índice
  • ✅ Menor uso de memoria
  • ⚠️ Necesita datos para entrenar
🌐

HNSW

Hierarchical Navigable Small World. Construye un grafo multicapa donde cada nodo conecta a sus vecinos más cercanos.

SQL 
CREATE INDEX ON documentos
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- m = conexiones por nodo (más = mejor recall)
-- ef_construction = calidad del índice
  • ✅ Mayor recall que IVFFlat
  • ✅ No necesita entrenar
  • ⚠️ Más memoria y tiempo de creación
