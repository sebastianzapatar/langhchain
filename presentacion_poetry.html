Gestión de Paquetes
con Poetry

La nueva forma moderna, determinista y segura de manejar entornos y dependencias en Python, dejando atrás el caos de requirements.txt.

¿Por qué no usar requirements.txt?

El Problema de requirements.txt

Normalmente, un pip freeze > requirements.txt genera una lista plana con cientos de dependencias cruzadas. Si instalas un paquete que requiere otro, ambos quedan en el archivo. Al desinstalar, la basura se queda. No hay forma de distinguir entre dependencias directas tuyas y sub-dependencias.

La Solución con Poetry

Poetry actúa como un Resolvedor de Dependencias profundo. Solo declaras lo que TÚ necesitas en un archivo estructurado. Poetry se encarga matemáticamente de encontrar las versiones compatibles de todas las sub-dependencias y aislar el entorno.

Ventajas vs Desventajas

Ventajas

  • Determinismo: Siempre instalas exactamente lo mismo gracias al lockfile.
  • Entornos Automáticos: Crea y gestiona el virtualenv por ti.
  • Empaquetado Fácil: Construir y publicar a PyPI con un solo comando.
  • Grupos: Separación clara entre dependencias de producción, desarrollo o testing.

Desventajas

  • Curva de Aprendizaje: Requiere entender nuevos comandos.
  • Resolución Estricta: Si hay paquetes viejos incompatibles, Poetry se rehusará a instalar en lugar de forzarlo (lo cual es bueno, pero frustrante al inicio).
  • Instalación Global: Requiere una instalación inicial fuera de tu entorno de proyecto.

Los Archivos Principales

Poetry se basa en el estándar PEP 518 para reemplazar setup.py y requirements.txt simultáneamente.

pyproject.toml

Es el corazón del proyecto. Contiene metadatos (nombre, autor), configuraciones de herramientas (como pytest o black) y las dependencias declaradas directamente por ti.

[tool.poetry.dependencies]
python = "^3.10"
fastapi = "^0.100.0"
langchain = "^0.1.0"

[tool.poetry.group.dev.dependencies]
pytest = "^7.0.0"

poetry.lock

Se auto-genera al ejecutar poetry install o poetry add. Contiene la resolución exacta de TODO el árbol de dependencias, incluyendo hashes de seguridad. ¡Debe subirse a Git!

[[package]]
name = "anyio"
version = "3.7.1"
description = "High level compatibility layer for..."
dependencies = { idna = ">=2.8", sniffio = ">=1.1" }

¿Cómo Instalarlo?

1. Instalación Global en tu Sistema

Poetry debe instalarse globalmente para que pueda gestionar múltiples proyectos. No lo instales con pip dentro de un virtualenv.

Mac / Linux

curl -sSL https://install.python-poetry.org | python3 -

Windows (PowerShell)

(Invoke-WebRequest -Uri https://install.python-poetry.org -UseBasicParsing).Content | py -

2. En un Proyecto Nuevo o Existente

Proyecto Nuevo poetry new mi_proyecto
Proyecto Existente poetry init
Clonar e Instalar poetry install

Ejemplos de Comandos y Ejecución

Gestionar Paquetes

  • poetry add requests
    Añade a prod y actualiza lockfile.
  • poetry add pytest --group dev
    Añade solo para desarrollo.
  • poetry remove requests
    Remueve y limpia sub-dependencias.

Correr la Aplicación

Poetry crea el entorno virtual de fondo. Para correr código dentro de ese entorno, tienes dos opciones:

  • 1. Correr prefijando poetry:
    poetry run python mi_script.py
  • 2. Activar la shell virtual:
    poetry shell
    (Y luego ejecutas normal: python mi_script.py)

Ejemplo en un Proyecto Real

Así se ve la estructura y flujo de trabajo de un proyecto backend profesional gestionado por Poetry.

Estructura Recomendada

mi_super_api/
├── pyproject.toml      # Meta y dependencias
├── poetry.lock         # Hashes de seguridad exactos
├── README.md
├── src/                # Código fuente principal
│   └── super_api/
│       ├── __init__.py
│       ├── main.py     # Instancia de FastAPI
│       └── routers/
└── tests/              # Entorno de pruebas
    ├── test_main.py
    └── conftest.py

Flujo de Trabajo Profesional

  • INIT poetry init --python ">=3.10,<4.0"
  • PROD poetry add fastapi uvicorn psycopg2
  • DEV poetry add --group dev pytest black
  • LOCK poetry lock (Asegura versiones)
  • RUN poetry run uvicorn src.super_api.main:app
  • TEST poetry run pytest tests/

Ejemplo: Nuestro Proyecto RAG

Así estructuramos la configuración de este mismo proyecto (LangChain + PostgreSQL + FastAPI).

Estructura del Repositorio

langhchain/
├── pyproject.toml      # Todas las dependencias juntas
├── poetry.lock         # Asegura entorno idéntico para todos
├── .env                # Credenciales (OPENAI_API_KEY, Postgres)
├── compose.yml         # PostgreSQL + pgvector Docker
├── local_rag/          # Agente RAG 100% Local (Ollama)
│   ├── agent.py
│   └── api.py
├── openai_rag/         # Agente RAG OpenAI
│   ├── agent.py
│   └── api.py
├── documentos/         # PDFs para embeber en la Vector DB
└── frontend.html       # Interfaz Web del Chat

Paso a Paso para Configurar Todo

  • 1. DB docker-compose up -d (Levanta Postgres)
  • 2. INIT poetry init --python ">=3.10,<4.0" (Crea pyproject.toml)
  • 3. ADDS poetry add langchain langchain-openai langchain-postgres "psycopg[binary]" fastapi uvicorn pypdf python-dotenv
  • 4. RUN L poetry run python -m local_rag.api (API Local)
  • 5. RUN O poetry run python -m openai_rag.api (API OpenAI)
  • 6. FRONT Abrir frontend.html o index.html
