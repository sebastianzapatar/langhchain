Sistemas RAG Autónomos y Privados

Construyendo un cerebro corporativo local con Ollama (Llama 3.1), PostgreSQL y LangGraph.

¿Qué es RAG?

Retrieval-Augmented Generation es una técnica que le da a un Modelo de Lenguaje (LLM) la capacidad de "leer" documentos privados antes de responder.

El Problema
Llama 3.1 es inteligente, pero no sabe nada de los documentos internos de tu empresa o de PDFs confidenciales.
El Retrieval (Búsqueda)
Buscamos en nuestra base de datos vectorial los párrafos más relevantes relacionados con la pregunta del usuario.
La Generación
Le pasamos esos párrafos al LLM y le decimos: "Responde al usuario basándote SOLO en esta información."

Nuestra Arquitectura Local

Hemos diseñado una solución 100% privada. Sin enviar datos a OpenAI ni a internet. Todo corre en la memoria de la máquina.

1. Ingesta (ETL)
PyPDFLoader + RecursiveSplitter
2. Vector DB
PGVector + Embeddings
3. Agente de IA
Llama 3.1 + LangGraph
4. API Rest
FastAPI (/chat)

Fase 1: Ingesta (ETL)

Los LLMs no pueden leer un PDF de 500 páginas de golpe (colapsaría su memoria de contexto). Por eso usamos la técnica del Chunking.

  • Extract: Leemos los archivos .txt y .pdf.
  • Transform: Partimos el texto en trozos de 1000 caracteres, solapándolos 200 caracteres para no cortar ideas (RecursiveCharacterTextSplitter).
  • Load: Los guardamos en PostgreSQL.
local_rag/ingestion.py
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200,
    add_start_index=True
)
chunks = text_splitter.split_documents(documentos)

vector_store = PGVector(
    embeddings=embeddings,
    collection_name="documentos_locales",
    connection=PG_URI
)
vector_store.add_documents(chunks, ids)

El Secreto: Embeddings

¿Cómo busca la base de datos el texto más relevante si no usamos palabras exactas? Usamos matemáticas espaciales.

nomic-embed-text
Este pequeño modelo local de Ollama lee cada "chunk" de texto y lo convierte en un Vector (una lista inmensa de coordenadas matemáticas).

Textos con significados similares terminan teniendo coordenadas cercanas en el espacio vectorial.
PGVector (PostgreSQL)
Nuestra base de datos tradicional de Postgres tiene una extensión que le permite guardar estos vectores y hacer Búsqueda de Similitud (Búsqueda Semántica).

Cuando el usuario pregunta, convertimos la pregunta a un vector y buscamos los vecinos más cercanos. 
local_rag/agent.py
@tool
def buscar_en_documentos(consulta: str):
    """Busca en la base vectorial"""
    retriever = obtener_retriever()
    return retriever.invoke(consulta)

Fase 2: El Agente Inteligente

No estamos haciendo una simple cadena estática. Construimos un Agente Autónomo usando LangGraph.

Le damos al modelo `llama3.1:8b` una herramienta (Tool): buscar_en_documentos.

El flujo dinámico:

  1. El usuario hace una pregunta.
  2. El LLM piensa: "¿Sé esto? No. ¿Debo usar la herramienta? Sí."
  3. El LLM ejecuta la búsqueda vectorial.
  4. Lee los resultados.
  5. Genera la respuesta final en español perfecto.

Memoria Conversacional

Los modelos de IA no tienen memoria por defecto; cada pregunta es una "hoja en blanco". Para que haya una conversación fluida, usamos LangGraph Checkpointers.

MemorySaver
Intercepta cada estado del grafo (los mensajes enviados y recibidos) y los guarda en RAM.
El Thread ID
Cada conversación tiene un identificador único. Si pasas el mismo ID, el agente "recuerda" la charla anterior.
El Contexto
Antes de responder, el sistema inyecta automáticamente todo el historial de la charla en el Prompt.

Fase 3: Exposición como API REST

Para que este motor de IA sirva de algo en el mundo real, necesitamos exponerlo para que los Frontend (Angular, React, Vue) puedan consumirlo.

POST /ingestar

Lee la carpeta local, vectoriza y guarda. Se ejecuta cuando un admin sube nuevos documentos.

POST /chat

Recibe un JSON {"mensaje": "Hola", "thread_id": "123"} y devuelve la respuesta del Agente + los pasos internos que tomó.
